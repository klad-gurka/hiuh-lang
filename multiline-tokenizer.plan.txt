Goal
hiuh-tokenizer.hiuh should process all lines of stdin, not just the first.
Each line is read with Läs, tokenized, then the next line is read.
Stop when stdin is exhausted (EOF).

Problem 1: No EOF detection
fgets returns NULL on EOF, but the current READ instruction discards the
return value. We need a way to detect EOF from HIUH code.

Chosen solution: modify READ to zero input_buf[0] before calling fgets,
then leave the fgets return value in a dedicated implicit variable `las_ok`
(1 = got data, 0 = EOF). The tokenizer can then check:
  Om las_ok är 0
      JagMåsteGåNu 0
  Hejdå

Compiler change (hiuh-native.py, READ case, Windows):
  Before fgets call:
      lea input_buf(%rip), %rcx
      movb $0, (%rcx)            # pre-zero first byte
  Call fgets as before.
  After fgets:
      test %rax, %rax
      setnz %al
      movzx %al, %rax
      mov %rax, {las_ok_reg}     # las_ok = 1 if got data, 0 if EOF
  las_ok is allocated as a normal variable via alloc_var('las_ok')
  when first referenced (same as träff).

Tokenizer change: add `Sätt las_ok till 0` before the outer loop to
force allocation, then check `Om las_ok är 0` immediately after `Läs`.

Problem 2: No outer loop construct
HIUH only has FOR (counted). We need to loop "until EOF".

Chosen solution: no new language feature. Wrap everything in a large
FOR loop (0 to 9999) and use JagMåsteGåNu 0 on EOF. This is the same
pattern already used for the inner character loop.

Problem 3: Variable count budget
Outer FOR adds one more variable (the outer loop counter, e.g. `rad`).
Current budget: antal(r12), j(r13), i_ord(r8), klar(r9), i(r10), träff(r11)
— all 6 GP registers used.

Adding `rad` (outer loop counter) and `las_ok` would overflow to 8 variables.

Solution: eliminate `rad` by noting the outer loop only exits via
JagMåsteGåNu — the counter is never read. Use a dummy name that still
compiles but we accept the register is wasted.

Actually better: reduce variables elsewhere:
  - Drop `träff` as an explicit variable — make CMP_BUF_LIT store its
    result directly into a fixed register (%r11) without going through
    alloc_var. Same approach as tecken → %r15.
  - This frees one GP slot for las_ok.
  - rad (outer loop counter) stays implicit in the FOR machinery and
    doesn't need to be read, so it still consumes a register but that
    is acceptable — just means one fewer user variable.

Register plan after change:
  antal  → r12
  j      → r13
  i_ord  → r8
  klar   → r9
  las_ok → r10  (allocated by first Sätt las_ok till 0)
  rad    → r11  (outer FOR counter, never read by program)
  i      → ??? OVERFLOW

Problem: adding an outer FOR loop (rad) plus the existing inner FOR (i)
plus las_ok = 7 GP variables + tecken in r15. Still one over budget.

Real solution: merge las_ok check into the EOF character detection.
On EOF fgets returns NULL and does NOT write to input_buf (pre-zeroed
first byte stays 0). So after Läs, the inner FOR loop immediately sees
tecken==0 at index 0, which already triggers JagMåsteGåNu. No las_ok
variable needed at all.

Pre-zero approach (no new variable):
  Compiler: before fgets call, emit `movb $0, input_buf(%rip)`
  On EOF: fgets returns NULL, input_buf[0] stays 0
  On normal line: fgets writes the line, input_buf[0] is first char
  The inner loop's existing `Om tecken är 0 → JagMåsteGåNu 0` fires on EOF.

But then we can't distinguish "empty line" from "EOF".
Empty line: fgets writes "\n\0", input_buf[0] = '\n' = 10 — not 0.
EOF: fgets returns NULL, input_buf[0] = 0 — fires JagMåsteGåNu.
This works correctly.

Final variable budget (outer FOR + inner FOR):
  antal  → r12
  j      → r13
  i_ord  → r8
  klar   → r9
  rad    → r10  (outer loop counter)
  i      → r11  (inner loop counter)
  träff  → special, stored in %r15b after CMP_BUF_LIT (like tecken)
  tecken → %r15 (already special-cased)

Wait — träff and tecken both want %r15. Conflict.

Resolution: träff result is used immediately (Jämför then Om träff är 1
with no intervening instructions). Store träff in %rax temporarily
after CMP_BUF_LIT, then the CMP instruction reads it from %rax.
Add a dedicated `cmp_result` pseudo-register (%rbx or a stack slot)
for the träff value, separate from %r15.

Or simpler: make träff a normal variable but drop one other variable.
The outer FOR counter `rad` is never read — can we avoid allocating it?

Yes: add compiler support for anonymous FOR counter. Syntax:
  För _ från 0 till 9999
The _ variable is never alloc_var'd, loop uses a scratch register
directly without consuming a named slot.

This brings the budget back to 6 named variables:
  antal, j, i_ord, klar, i (inner), träff = 6 ✓
  rad → anonymous, uses a scratch register not in var_reg

Part 1: Compiler changes
1a. READ: pre-zero input_buf[0] before fgets call (Windows + Linux)
1b. FOR with anonymous counter: `_` as loop var skips alloc_var,
    uses %rax or a scratch not in the GP variable pool

Part 2: Tokenizer changes
  Outer loop: För _ från 0 till 9999
      Läs
      (inner character loop — existing logic, unchanged)
      Sätt j till 0       ← reset word buffer between lines
      Sätt klar till 0    ← reset state between lines (already happens)
  Hejdå

  The JagMåsteGåNu 0 inside the inner loop on tecken==0 exits the
  whole program on EOF. On normal EOF the inner loop fires it.
  No other changes to inner loop logic needed.

Part 3: Implementation order
1. Add input_buf[0] pre-zero to READ in hiuh-native.py
2. Add anonymous loop counter `_` support to FOR tokenizer + compiler
3. Wrap hiuh-tokenizer.hiuh inner loop in outer För _ från 0 till 9999
4. Add Läs + reset at top of outer loop body
5. Test: echo -e "Sätt x till 0\nFör i från 0 till 5\nHejdå" | hiuh-tokenizer.exe
   Expected: SET x till 0 / FOR i från 0 till 5 / END / (count)
